## Pass real recommendation impression IDs to feed cards
labels: type:bug, area:analytics, priority:p1

**File:** src/lib/repositories/user-data.ts:1050

`getFeedPageData` currently maps ranked papers into `FeedPaper` objects with `recommendationImpressionId: undefined`, then records recommendation impressions later inside `after(...)`. The client-side deck mutations only send a `recommendationImpressionId` when one is present, and `/api/deck` only attaches attribution after validating that ID.

Consequence: feed actions such as dismiss, favorite, read-later, and open-detail can be recorded with a null `recommendation_impression_id`, even though the impression was generated from the feed. This weakens analytics, feedback attribution, future ranking diagnostics, and any offline evaluation of recommendation quality.

**Fix:** record the visible recommendation impressions before returning the feed payload and attach the returned IDs to each `FeedPaper`. If doing that synchronously becomes too expensive, pre-generate the impression IDs in the response and insert the corresponding rows immediately, or move impression creation into the first deck mutation using a batch/rank token that can be validated server-side.

## Restore prefetch=false on authenticated navigation links
labels: type:bug, area:performance, priority:p1

**File:** src/components/app-shell.tsx:57

The task log says authenticated navigation prefetch was disabled after production HAR traces showed extra RSC requests around feed actions. The current `AppShell` desktop links and mobile settings/logo links do not pass `prefetch={false}`. The current `BottomNav` links also omit it.

Consequence: in production, visible authenticated links may trigger background route/data prefetching again. On dynamic, user-specific pages (`/feed`, `/library`, `/settings`, `/digest`, `/search`), this can reintroduce avoidable network traffic and make mobile taps feel slower or noisier to debug.

**Fix:** centralize authenticated navigation in a small `AppNavLink` component that sets `prefetch={false}` by default, and use it in both `AppShell` and `BottomNav`. Add a focused Playwright/HAR smoke check or regression note for feed actions so this does not silently regress again.

## Apply already_read feedback when building ranking and profile weights
labels: type:bug, area:recommendations, priority:p2

**File:** src/lib/ranking/feed-ranking.ts:42

The detail feedback route records the action `already_read`, and `isFeedHiddenAction` correctly hides papers with that action from the feed. However, both ranking feedback weights and profile-embedding interaction weights include the older `read` action but not `already_read`.

Consequence: clicking `Already read` removes the specific paper from the feed, but it does not positively train the user's topic feedback or profile embedding. This makes one of the strongest explicit relevance signals weaker than intended.

**Fix:** either add `already_read` to the positive interaction weights in both `feed-ranking.ts` and `user-profile-embeddings.ts`, or consolidate the enum/actions so the UI records `read` consistently. If both values are kept for compatibility, document their semantics and add unit tests that `already_read` affects feedback weights.

## Make settings interest autosave ordered and recoverable
labels: type:bug, area:settings, priority:p2

**File:** src/components/settings-interest-editor.tsx:133

Each interest toggle immediately calls `saveSettingsInterestsAction(Array.from(next))`. Rapid taps can create overlapping full-replacement mutations. The action rewrites the whole selected-topic set, refreshes the profile embedding, clears feed recommendations, and revalidates multiple pages. The client does not sequence requests, cancel stale saves, rollback on failure, or show an error message.

Consequence: an older request can finish after a newer one and overwrite the user's final selection. On failure, the UI may keep showing an unsaved state while the server keeps the previous state.

**Fix:** replace immediate autosave with an explicit `Save changes` button, or debounce and serialize mutations with a monotonic request id. While a save is in flight, either disable further toggles or keep a pending queue. Surface failures with `MutationAlert` and rollback to the last confirmed server state.

## Upgrade catalog search from broad ILIKE scans to indexed relevance search
labels: type:enhancement, area:search, priority:p2

**File:** src/lib/repositories/catalog.ts:238

`searchPapers` currently uses a `%query%` `ILIKE` pattern over title, abstract, venue, arXiv ID, DOI, author names, and topic labels, then orders matches by year and citation count. This is acceptable for a small catalog, but it will become increasingly slow and low-signal as the corpus grows.

Consequence: search UX will not scale well with larger arXiv backfills. Users also get recency/citation-biased ordering rather than match-quality ordering, so exact title/author/topic hits may not reliably appear first.

**Fix:** add a lightweight indexed search layer before considering full-text/RAG. A good free-first option is Postgres `tsvector` with weighted fields (`title`, `authors`, `topics`, `abstract`) plus a GIN index, optionally combined with `pg_trgm` for fuzzy title/author matching. Return a rank score and expose simple match context in the UI. This stays separate from full-text PDF ingestion.

## Show users when a new PWA version is ready
labels: type:enhancement, area:pwa, priority:p3

**File:** src/components/pwa-provider.tsx:13

`PwaProvider` registers the service worker and detects `updatefound`, but when a new worker is installed it only logs `Update available â€” refresh to activate` to the console. Mobile/PWA users will never see this signal.

Consequence: installed users can remain on stale assets until they manually reload or the browser eventually refreshes the app. This is especially confusing after UI fixes because the app gives no indication that a newer build is available.

**Fix:** expose a small in-app update toast/banner when `worker.state === "installed" && navigator.serviceWorker.controller`. The CTA can say `Update available` / `Refresh`, then call `window.location.reload()` after user confirmation. Keep dynamic app data network-only; this issue only concerns the update lifecycle UX.

## Harden LaTeX rendering for common arXiv delimiters and escaped dollars
labels: type:enhancement, area:ui, priority:p3

**File:** src/lib/render-latex.ts:5

`renderLatex` only tokenizes `$...$` and `$$...$$`. It does not recognize common delimiters such as `\(...\)` and `\[...\]`, and it does not treat escaped dollars (`\$`) specially.

Consequence: some arXiv abstracts can show raw LaTeX delimiters, while text containing escaped dollar signs may be incorrectly consumed as inline math. Since PaperDeck surfaces abstracts directly in feed/detail/search cards, this can produce visibly broken scientific text.

**Fix:** replace the simple `indexOf("$")` scanner with a delimiter-aware tokenizer that handles `\(...\)`, `\[...\]`, `$$...$$`, `$...$`, escaped delimiters, and unbalanced delimiters. Add unit tests for escaped dollars, unmatched math delimiters, display math, and mixed text/math fragments.

## Add active state to desktop navigation
labels: type:enhancement, area:ui, priority:p3

**File:** src/components/app-shell.tsx:57

The mobile `BottomNav` uses `usePathname()` and visually marks the active section. The desktop navigation in `AppShell` is static and does not set an active style or `aria-current`.

Consequence: on wider screens, users have weaker spatial orientation than on mobile. This is noticeable because PaperDeck has multiple persistent authenticated surfaces: feed, digest, search, library, and settings.

**Fix:** extract the desktop nav into a client component or pass the current path from a small client wrapper. Reuse the same active-state logic as `BottomNav`, including the `/papers/...` â†’ Feed mapping, and add `aria-current="page"` for the active item.

## Evaluate Motion for the deck swipe state machine after correctness fixes
labels: type:enhancement, area:ui, priority:p3

**File:** src/components/feed-deck.tsx:120

The feed deck currently implements a custom pointer state machine for drag start, drag move, pointer capture, threshold detection, exit animation, and delayed mutation dispatch. This works, but it spreads gesture behavior across local refs, timeout timing, and manual transform strings.

Consequence: future refinements such as velocity-sensitive swipes, reduced-motion behavior, direction locking, spring physics, and consistent rollback animation will be harder to maintain by hand.

**Fix:** after the existing swipe persistence bugs are fixed, evaluate whether a small gesture/animation layer such as Motion for React is worth the added dependency. Keep the current custom implementation if bundle size or dependency constraints matter more, but document that choice and add interaction tests for swipe-left and swipe-right behavior.
